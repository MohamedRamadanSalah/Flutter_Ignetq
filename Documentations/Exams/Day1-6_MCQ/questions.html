Day 1 — Clean Architecture & the Dependency Rule

1.codemediumAnalyze
A junior pushes this file, located at domain/repositories/task_repository.dart. The CI architecture-lint fails. Beyond "it imports Dio," what is the deepest reason this is wrong?
import 'package:dio/dio.dart';
import '../entities/task.dart';

abstract interface class TaskRepository {
  Future<Response> getTasks({String? cursor});
}
  1. The contract returns Response instead of Either<Failure, T>, so it cannot represent failures as values.
  2. abstract interface class is invalid for a repository; only abstract class may be implemented across layers.
  3. The domain's contract now depends on a network library, so the business rules can no longer be swapped onto a non-Dio transport — the abstraction is no longer an abstraction.
  4. The import path uses ../entities/task.dart instead of an absolute package import, breaking the dependency rule.

9.conceptualmediumApply
Why is returning Either<Failure, List<Task>> from the repository considered superior to letting it throw NetworkException for the use case to catch?
  1. Either is faster at runtime because it avoids the cost of building stack traces.
  2. Throwing forces every caller to remember a try/catch and lets raw, untyped exceptions leak across the boundary; Either makes failure an explicit, typed value the caller must handle in the type system.
  3. Either allows the repository to return multiple successful values at once, which exceptions cannot.
  4. Exceptions cannot cross an async boundary in Dart, so Either is technically required here.

Day 2 — Networking & the API Client

18.codehardAnalyze
buildDio() registers interceptors in this order. A reviewer rejects it. What concrete, observable defect does this ordering produce?
dio.interceptors.add(LogInterceptor(requestBody: true));   // 1
dio.interceptors.add(AuthInterceptor(() => readToken()));  // 2
dio.interceptors.add(ErrorInterceptor());                  // 3
  1. The request log is written before the Authorization header is added, so logs show requests missing the Bearer token even though it was actually sent.
  2. The ErrorInterceptor will fire before the AuthInterceptor on the response path, so 401s are mapped before a token is ever attached.
  3. LogInterceptor and AuthInterceptor both run onRequest, so the token gets added twice and the server rejects a malformed header.
  4. Dio runs interceptors in reverse registration order on the request path, so auth fires first anyway and the ordering is harmless.
19.scenariomediumEvaluate
On a flaky metro signal, a GET /tasks and a POST /tasks both fail with receiveTimeout. Your retry interceptor retries any request on timeout, twice. The senior objection?
  1. The policy is correct — both GET and POST are idempotent by HTTP definition, so retries can never cause duplicates.
  2. Retrying either is unsafe because timeouts always mean the server never received the request, so a retry just doubles load for nothing.
  3. Retrying the GET is fine, but retrying the POST risks creating duplicate tasks because a receiveTimeout can occur after the server already processed and committed the write.
  4. The GET retry is the dangerous one, because reading the same resource twice can return stale data and corrupt the local cache.

    final newToken = await _refresh();          // POST /auth/refresh
    final opts = err.requestOptions;

96.conceptualhardEvaluate
Why does Either<Failure, T> "beat throwing," beyond style?
  1. Either is faster at runtime because it avoids the cost of building stack traces.
  2. Throwing is illegal in Dart's null-safe mode, so Either is the only option that compiles.
  3. Either cannot represent success, so it eliminates the ambiguity of partial results.
  4. Either turns the failure into a value the type system forces the caller to handle (both Left and Right), whereas a thrown exception relies on the caller remembering to catch — a silent omission that crashes at runtime.
